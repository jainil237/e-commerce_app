# Follow-Up Policy

Follow-ups capture work that should not be hidden in the reflection narrative.

Required fields:

- action
- owner
- suggested artifact, ticket title, PR, release action, source update, or lifecycle phase
- status: open, done, blocked, deferred, or waived
- affected manifest IDs when applicable

Follow-up types:

- new brief
- plan revision
- Build fix
- Review/Test rerun
- release or rollback action
- source-of-truth update
- skill/template/config/schema/validator improvement
- learning curation request

Rules:

- Every deferred or blocked active requirement needs a follow-up unless explicitly waived.
- Follow-ups must be specific enough to start the next action.
- Do not create external tickets unless the user requested it and a configured tool is available.
- If the follow-up needs user decision, name the exact question.
