# Explicit Requirements

Explicit requirements become `R` IDs.

An item is explicit when:

- the user directly asks for it
- the source item directly requires it
- a correction from the user changes scope or acceptance
- the user explicitly approves a waiver, exclusion, or non-goal

Rules:

- Keep `R` wording behavior-level and outcome-focused.
- Do not encode exact code shape unless the user asked for that shape.
- Give every `R` at least one `Acceptance:` bullet.
- Split separate outcomes into separate `R` IDs.
- Preserve existing `R` IDs even when wording is clarified; append new IDs for new requirements.

Reject vague acceptance such as "works" or "is done". Acceptance must describe observable proof.
